# netkeirin「ウマい車券」入稿フォーム API仕様（解析結果）

調査日: 2026-07-23（初回解析）/ 2026-07-23夜（追加実機検証・bet_id実データ確定）
調査方法: Chrome DevTools相当のライブトラフィック計測（javascript_toolによるXHR/fetchのmonkey-patch）。
静的リソース（IpatDataModel.js等）はサイズ超過・単一行JSONで静的解析が困難だったため、
実際のフォーム操作で発生したリクエストを直接キャプチャして確定させた。
実機検証は 富山9R・富山10R・松阪8R（テスト）・佐世保1R（2026-07-24分、bet_id確定用）で実施。
テスト入稿は全てサーバー送信前にlocalStorageからクリア済み・未公開・アカウントはクリーンな状態。

**重要な訂正（2026-07-23夜）**: 当初「houshiki は2桁コード（例 `'86'`）」と推定していたが、
実際に `race.html` の買い目編集UI（`ipat_sp.html`）を操作してブラウザの `localStorage`
(`ndi::umaibet_{race_id}`) を直接読み取ったところ、shikibetu と houshiki は
**別トークン**（`_b8_c6` のように分離）であることが確定した。3.節を参照。

**さらに重要な訂正（2026-07-23深夜・実装後の実機テストで発覚）**: ログイン成功後に
`race.html`（認証必須ページ）の実ソースを直接取得できたことで、以下が確定した
（コード実装・テスト提出・削除まで全て実施済み）:
- `type` パラメータは式別(shikibetu)ではなく**勝負アイコン値**(`$('#act-type').val()`)。
- `point` パラメータはポイント数ではなく**販売価格**(`$('#act-point').val()`)。
- 式別/方式は `kaime[].bet_id` 文字列にのみ含まれ、top-levelパラメータには存在しない。
- `mark` の値は数値ではなく**文字列**("1"/"2"/"4"等)。DOM id分割由来のためstring型。
- ログインエンドポイントは `auth/api_post_login.html`（`login/api_post_login.html`ではない）、
  フィールド名は `user_id`/`password`（`login_id`ではない）。詳細は7節参照。
2.1節・2.3節は訂正済みの内容に更新している。

対象サイト: `https://tool.syakenv2.netkeiba.com/bettool/`（netkeirin公式「ウマい車券」予想入稿ツール）

## 1. 全体フロー

```
1. ログイン (api_post_login.html) → セッションCookie発行
2. race_auth.html で公開待ち一覧 / 入稿対象レース選択
3. race.html?race_id=... で予想入力（◎○▲△☆マーク、買い目、コメント）
4. api_post_goods.html へ POST (action=add) → 下書き保存（公開待ちに反映）
5. ユーザーが目視確認の上「公開」ボタンを押下 → 初めて一般公開される
```

**重要**: ステップ4（下書き保存）まではプログラムから自動化可能。ステップ5（公開）は
必ず人間が最終確認の上クリックする運用とする（自動化対象外）。

## 2. 入稿エンドポイント

```
POST https://tool.syakenv2.netkeiba.com/bettool/bet/api_post_goods.html
Content-Type: application/x-www-form-urlencoded
Cookie: (ログイン済みセッションCookie必須)
```

### 2.1 パラメータ一覧

| パラメータ | 型 | 内容 |
|---|---|---|
| `output` | 固定値 | `"json"` |
| `action` | 文字列 | `add`（新規/upsert）。他に `get`（現状取得）・`update`（既存更新）・`delete`（削除、item_id必須）がある |
| `race_id` | 文字列 | `YYYYMMDD` + 場コード2桁 + レース番号2桁（例: `202607234708`） |
| `mark` | JSON文字列 | 車番(文字列キー) → mark_code(**文字列値**)のマップ（下記2.2） |
| `title` | 文字列 | 予想タイトル（**必須・空文字不可**、script/style/iframeタグ禁止） |
| `comment` | 文字列 | コメント本文（**必須・空文字不可**、script/style/iframeタグ禁止） |
| `type` | 文字列 | **勝負アイコン値**（`0`=指定しない/`1`=自信あり/`2`=穴狙い）。式別ではない（要訂正済み） |
| `point` | 文字列 | **販売価格**（`100`〜`900`、100刻み）。ポイント数ではない（要訂正済み） |
| `waku_check` | JSON配列 | 2車以上が同枠になる枠番のリスト（下記2.4） |
| `kaime` | JSON配列 | `{bet_id, bet_money}` のリスト（下記2.5）。式別/方式はbet_id文字列にのみ含まれる |

`action=delete` の場合は `race_id`/`mark`等は不要で `item_id`（`race_auth.html`の削除
ボタン`id="act-yoso_delete_{item_id}"`から取得、例: `b1723163_732`）のみ必須。
成功時レスポンス: `{"status":"OK","item_id":"..."}`。

### 2.2 mark_code（マーク）

| コード | マーク | 確認状況 |
|---|---|---|
| 1 | ◎ | 確認済み（実データ） |
| 2 | ○ | 確認済み（実データ） |
| 3 | ▲ | 確認済み（2026-07-23夜、race.html の印選択UI `#act-mark_{車番}_3` のラベルで確認） |
| 4 | △ | 確認済み（実データ） |
| 5 | ☆ | 確認済み（同上 `#act-mark_{車番}_5`） |
| 0 | --（印なし） | 確認済み（デフォルト値） |

今回運用する「二軸探偵」方式では ◎(1着軸=1番手指数) / ○(2着軸=2番手指数) / △(総流し=残り全馬)
の3種のみ使用する。mark_code は `race.html?race_id=...` の印プルダウン
（`id="act-mark_{車番}_{mark_code}"`）のDOM順序と完全一致することを確認済み。

### 2.3 式別（shikibetu）・方式（houshiki）【2026-07-23夜: 実データで確定】

`race.html` → 買い目編集画面（`ipat_sp.html?race_id=...`）で実際に
「3連複」→「ながし」→「軸2」→ 軸1=1・軸2=2・相手=3,4,5,6,7・各2,000円 を選択し、
`localStorage.getItem('ndi::umaibet_{race_id}')` を直接読み取って確定：

```json
{"list":{"...":{"bet_id":"a5-85-1_b8_c6_1_2_3-4-5-6-7","bet_money":2000}},"uptime":"..."}
```

- `b8` = shikibetu 8 = **3連複**（確認済み）
- `c6` = houshiki 6 = **軸2頭ながし**（確認済み）。houshiki は shikibetu と結合した2桁コードではなく
  **独立したトークン**（`_b{shikibetu}_c{houshiki}`）である。旧版の「`'86'`」という推定は誤りだったため訂正。
- 券種選択UIには 2車複/2車単/3連複/3連単/ワイド/2枠複/2枠単 があり、方式選択には
  通常/フォーメーション/ボックス/ながし があった。「ながし」を選ぶと更に「軸1」/「軸2」の
  サブ選択が現れる（今回使うのは軸2）。houshiki の他の値（通常=?, ボックス=?, フォーメーション=?,
  軸1ながし=?）は今回未検証（必要になれば同様の手順で確認可能）。
- `bet_money` はUI上の金額入力欄（内部的に「入力値×100」で円換算、例: 入力"20"→2,000円）の
  **最終円換算後の値がそのままJSON化**される（2,000）。

### 2.4 waku_check

同一枠に2車以上が入っている枠を列挙するJSON配列。
kiseki側では `keirin.wt_entries.frame_no` を車番でグルーピングすれば導出可能なため、
スクリプト側での特別なロジックは不要（GROUP BY frame_no HAVING COUNT(*) >= 2）。

### 2.5 kaime（買い目本体）

```json
[
  { "bet_id": "a{曜日}-{場コード}-{レース番号}_b{shikibetu}_c{houshiki}_{軸1}_{軸2 or 空}_{相手番-区切り}", "bet_money": 100 }
]
```

- `軸2` が存在しない買い方（例: 単独軸ながし）の場合は空文字
- 相手番は複数車番をハイフンで連結
- 実測では「◎○軸2頭ながし・残り総流し」の場合、`kaime` はサーバー側が組み合わせを展開する
  前提の単一bet_id文字列のみで足りることを確認済み（クライアント側で全組み合わせを
  展開して送る必要はない）

## 3. bet_id 文字列フォーマット詳細【2026-07-23夜: 実データで確定】

```
a{曜日1桁}-{場コード2桁}-{レース番号(先頭ゼロ無し)}_b{式別}_c{方式}_{軸1車番}_{軸2車番}_{相手車番(ハイフン区切り)}
```

実データ例（2026-07-24・佐世保1R・軸1=1・軸2=2・相手=3,4,5,6,7・各2,000円）:
```
a5-85-1_b8_c6_1_2_3-4-5-6-7
```
→ 3連複・軸2頭ながし（◎1・○2）・相手 3,4,5,6,7 総流し

**注意点（旧版からの訂正）**:
- レース番号は race_id 内では2桁ゼロ埋め（`...8501`）だが、bet_id 内では**ゼロ埋めなし**（`1` であり `01` ではない）
- houshiki は shikibetu と別トークン（`_b8_c6`）。2桁結合ではない
- 軸2が存在しない買い方（軸1本ながし等）での区切り方は今回未検証

**曜日コード（a{曜日1桁}）**: 2026-07-24（金曜日）で `a5` を実測確認。
Python の `date.isoweekday() % 7`（月=1…土=6・日=0）と JS の `Date.getDay()`（日=0…土=6）は
月〜土（1〜6）で完全に一致するため、この実測1点だけで月〜土は曜日コードが確定する
（`isoweekday() % 7` または `getDay()` をそのまま使えばよい）。日曜日のみ2つの規約が
分岐しうる（ISO=7 or JS=0）ため、**日曜開催レースでの初回入稿時は要目視確認**。

## 5. keirin側での実装方針（2026-07-23 決定事項）

**実装場所は kiseki リポジトリではなく `keirin` リポジトリ**（推奨SS+/SS/S判定・
`wt_races`/`wt_entries`/`picks_history` は全て keirin リポジトリの守備範囲のため）。本ドキュメントは
初期調査を kiseki 側セッションで行った経緯で kiseki 側にも複製されているが、正本は keirin 側
（`keirin/docs/netkeirin-input-api-spec.md` にも同期すること）。

- 対象データソースは `scripts/notify_prerace_wt.py` 等が生成する候補JSON
  （`data/picks/wave_picks_wt_{date}_s4_candidates.json` / `_night_s4_candidates.json`）
  または `picks_history` テーブル（`route='wt'` かつ `rank='SEVEN_S4'`）。
- **対象抽出条件**: `gate_label ∈ {SS+, SS, S}`。gate_label は候補生成時点
  （`src/cli/main.py` の `wave-picks-wt` 内、`wt_overlap_n` + `axis1_class`/`axis2_class` から
  `s4_gate_label()` で算出）で既に確定できる値であり、`notify_prerace_wt.py` のT-15分判定を
  待つ必要はない（軸カー1台以上が出走取消になるリスクはあるが、下書き保存のみで人間の公開前
  確認を経るため許容）。
- **トリガー設計（2026-07-23 決定）**: 朝バッチ（`daily_picks_wt.sh`、cron 8:00）・
  夕バッチ（`evening_picks_wt.sh`、cron 16:00、`s4_evening_reselect.py` 確定後）それぞれの
  候補生成直後に、その時点のcandidate JSONから gate_label∈{SS+,SS,S} を抽出して都度netkeirinへ
  下書き入稿する。同一race_idへの再送信は上書きされる（1節参照）ため、朝夕で重複しても無害。
  各バッチの入稿完了後、該当バッチで新規入稿したレース一覧をまとめて**1本のDiscord通知**
  （日付・午前/午後・件数・確認用URL）を送信する。
- **軸1(◎)=axis1・軸2(○)=axis2・残り全馬=△(mark_code=4)**。買い目は3連複(`b8`)・軸2頭ながし
  (`c6`)・各2,000円・全通り。
- **勝負アイコン**: gate_label が SS+/SS の場合は `自信あり`(act-type=1)、S の場合は
  `指定しない`(act-type=0)。
- **販売価格**: デフォルト300のまま変更しない。
- **商品タイトル・コメントの内容**は別途検討（ユーザー方針）。当面は
  `"{場名}{R}R 二軸探偵"` 程度の機械的なプレースホルダで実装し、後日ルール確定後に差し替える。
- 入稿済みレースの追跡には新規テーブル（またはJSONログ）で `race_key` ごとに送信済みフラグを
  持たせ、Discord通知の集計・重複防止に使う。
- 認証情報は `.env` に格納（kiseki/keirin既存規約通り、コードにハードコードしない）。
  変数名: `NETKEIRIN_LOGIN_ID` / `NETKEIRIN_PASSWORD`。
- 最終「公開」ボタンは常に人間が操作（自動化しない） — ユーザー自身が確認用URLから確認の上、公開する。
- **実行場所はVPS**（`notify_prerace_wt.py` 等と同じ実行環境・DBアクセスにSSHトンネル不要のため）。

## 7. ログイン仕様・実装確定（2026-07-23深夜・実装後の実機テストで確定）

`src/netkeirin_client.py` 実装後、VPS上で実際に `NetkeirinClient().login()` を実行し
成功したことに加え、認証済みセッションで以下のページの実ソースを直接取得して確定させた
（`auth/login.html`・`bet/race.html?race_id=...`・`bet/race_auth.html`）。

### 7.1 ログイン

```
POST https://tool.syakenv2.netkeiba.com/bettool/auth/api_post_login.html
data: {"output": "json", "action": "login", "user_id": "<ID>", "password": "<PW>"}
成功レスポンス: {"status":"OK","user_id":"732"}（732は内部ユーザーID、アカウント固有）
```

未ログイン状態で `top/index.html` 等にGETすると `auth/login.html`
（タイトル「ログイン画面」）へ自動リダイレクトされる。**注意**: このログイン画面自体の
HTML内にも"ログアウト"という文字列が(共通ヘッダ由来で)含まれているため、
「本文に"ログアウト"を含むか」だけで認証判定するとfalse positiveになる。
最終URLが `auth/login.html` でないことも合わせて確認する必要がある
（`src/netkeirin_client.py::_is_logged_in()` で対応済み）。

### 7.2 type / point パラメータの正体（重要な訂正）

`race.html` の実ソース（`$('#act-submit').on('click', ...)` 内）で確認:
```js
param.type = $('#act-type').val();   // 勝負アイコン: 0=指定しない/1=自信あり/2=穴狙い
param.point = $('#act-point').val(); // 販売価格: 100〜900(100刻み)
```
式別(shikibetu)・方式(houshiki)はtop-levelパラメータには存在せず、`kaime[].bet_id`
文字列にのみ含まれる（3節参照）。2.1節の初版記載「type=式別・point=ポイント数」は
誤り（実際に送信を試みる前の推測に基づく誤認識）だったため訂正済み。

### 7.3 クライアント側必須バリデーション（`check_goods_data()`）

- `title`・`comment` は空文字だとエラー（「商品タイトルが入力されていません」等）
- `mark` は ◎(mark_code=1) がちょうど1つ必須（0個/2個以上はエラー）
- 合計買い目金額(`bet_money_sokei`)は1,000円以上10,000円以下（「二軸探偵」方式の
  5点×2,000円=10,000円は上限ぴったりで許容範囲内）
- `mark` の値は文字列型（DOM `id="act-mark_{車番}_{code}"` を`.split('_')`した文字列が
  そのまま使われるため、`{"1": "1", "2": "2", ...}` のように**文字列**で送る必要がある。
  数値で送ると `switch` 文の厳密比較(`case '1':`)に一致せず、バリデーション不整合の
  原因になりうる）

### 7.4 削除エンドポイント（テスト後のクリーンアップ用）

`race_auth.html`（公開前予想一覧）の削除ボタン実装から確認:
