3-3. DB書き込み方式（2026-06-20 VPS移行後）

原則: VPS では全スクリプトが KEIRIN_DB_URL（crontab で常設）により PostgreSQL へ直接書き込む（get_connection() が PG モードになる）。SQLite一次＋同期は旧 Mac 運用の方式で、VPS では使われない。
書き込み元方式対象
収集・候補・採点（全バッチ） get_connection() → PG 直接 upsert wt_races, wt_entries, wt_odds, picks_history 等
発走15分前 notify_prerace_wt.py 直接更新 + decisions JSON 永続化 picks_history.prerace_gami / gap23 / pred_combo
（旧）SQLite→PG 同期 migrate_sqlite_to_pg.py — VPS上の SQLite は 2026-06-20 で停止した遺物のため、stale ガード（SQLite が PG より2日以上古ければ skip・exit 0）でスキップされる（b685a31）。意図的な再同期は --force —

4-2. 合成オッズ・gap23（API計算/表示）

kiseki APIは wt_odds_snapshot (morning) から 合成オッズ = 1 / Σ(1/各目オッズ) を計算して返す。発走前実測（prerace_gami）・gap23（モデル2位-3位確率差）と並べて表示: 合成 8.2倍 / 直前 7.8倍✓ / g23 1.4pt

4-3. サマリー集計条件（2026-07-07 変更）

集計は採点結果基準に統一。以下をすべて満たすものだけ集計:

  • NOT miwokuri
  • bet_amount > 0（＝発走15分前判定で購入が確定したもの）
  • rank IN ('7PLUS_SS', '7PLUS_S')
旧仕様の「prerace_gami≥閾値で再フィルタ」は撤廃。ガミ判定は採点時に bet_amount/miwokuri へ反映済みのため、二重フィルタが SS（カット買い）を誤って除外する問題があった。SS/S ランク別内訳は model_evaluation（HOLD精度）と実運用集計の両方を表示。

5. ランク体系と購入条件（2026-07-03〜現行）

5-1. 対象と候補化（朝）

  • 7車立てレースのみ（2026-07-03〜。9車はS束ROI 94.8%・12ヶ月中6ヶ月赤字で除外。7車はROI 149.1%）
  • gap12（モデル予測確率の1位-2位差）≥ 0.07 を候補化（#CAND）。買い目は 2軸（pivot1,2）× 3着候補の三連複。

5-2. 発走15分前の確定判定（購入フィルタ3条件 → ランク）

順序フィルタ / ランク条件根拠（最終オッズ上限値）
① ガミ回避 三連複オッズ < 7.0倍 の目を除外。有効目ゼロなら見送り min≥7帯 ROI 2.18 [CI 1.85-2.51]・見送り帯[3,7)は ROI 0.80 で回収不能（14,892R）
② 合成オッズ 有効目の合成オッズ < 8.0 なら見送り ②③で 6月バックテスト ROI 130.2%→141.8%（対象-13%・除外分は63.3%）
③ gap23 モデル2位-3位の確率差 < 1.0pt なら見送り
④ 7PLUS_SS ガミカット後の有効目が 1〜3目 → その目のみ購入（各100円） 点数集中・高効率
⑤ 7PLUS_S 有効目4目以上 ∧ gap12 ≥ 0.10 → 有効目全点購入（ガミ目除外） 確信度高・面で回収
Aランクは廃止（2026-06-28）: gap12[0.07,0.10) の低確信帯は購入対象外。歴史バックテスト全期間 ROI 137.2%（2025-07〜2026-06前半・全12ヶ月黒字・n=10,755）は最終オッズによる上限値であり、実運用ROIは 2026-08 に2ヶ月分の実測で本評価予定。

5-3. 発走前判定に記録する統計（2026-07-08〜・採用判断のためのデータ蓄積）

  • score_sd / score_gap2r / score_mean — 競走得点統計。SD大＝軸堅い（歴史: 買いレースで sd≥Q1 残しは ROI 2.87→3.0-3.6）
  • pred_sd / pred_top2sum — 指数（モデル予測）統計。配当予測は得点統計より強い（低配当AUC 0.637）
  • いずれも現時点ではフィルタに未使用。live実測の蓄積後に採用判断。
